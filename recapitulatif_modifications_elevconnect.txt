1. VUE resources/views/commandes/create.blade.php
-------------------------------------------------------------------------------

PROBLEME RENCONTRE :
  ParseError - Unclosed '[' on line 54 does not match ')'

CAUSE REELLE :
  La directive Blade @json(...) sépare ses arguments (valeur / options /
  profondeur JSON) avec un simple explode(','), sans tenir compte des
  virgules imbriquées dans un tableau ou un appel de fonction PHP.

  Le code original passait directement un tableau construit à la volée :

      @json($annonce->reductions->map(fn($r) => [
          'min' => (int) $r->quantite_min,
          'max' => (int) $r->quantite_max,
          'pct' => (float) $r->pourcentage_reduction,
      ]));

  Chaque virgule séparant les entrées du tableau ('min' => ..., 'max' => ...,
  'pct' => ...,) était interprétée à tort par @json() comme une séparation
  entre ses propres arguments, ce qui cassait la compilation Blade -> PHP.

CORRECTION APPLIQUEE :
  Le tableau est désormais précalculé dans une variable PHP "plate" via un
  bloc @php ... @endphp, puis seule cette variable (sans virgule au niveau
  de l'expression) est passée à @json() :

      @php
        $tranches = $annonce->reductions->map(fn ($r) => [
            'min' => (int) $r->quantite_min,
            'max' => (int) $r->quantite_max,
            'pct' => (float) $r->pourcentage_reduction,
        ])->values();
      @endphp
      const tranches = @json($tranches);

  Aucune autre partie du fichier n'a changé (formulaire de commande, calcul
  JS du montant, styles). Le contrôleur (CommandeController.php) et le
  modèle (Commande.php) n'ont pas eu besoin d'être modifiés : le bug était
  strictement localisé dans la vue.

REGLE A RETENIR :
  Ne jamais passer à @json(...) une expression contenant une virgule au
  niveau supérieur (tableau littéral, plusieurs arguments...). Toujours
  précalculer dans une variable PHP intermédiaire avant de l'injecter.

2. VUE resources/views/commandes/show.blade.php
-------------------------------------------------------------------------------

PROBLEME RENCONTRE :
  Class "SimpleSoftwareIO\QrCode\Facades\QrCode" not found
  (puis, après une première tentative de correction :
   Class "F9WebLtd\QrCode\Facades\QrCode" not found)

CAUSE REELLE :
  Le package original simplesoftwareio/simple-qrcode est un dépôt à
  l'abandon. Il dépend de bacon/bacon-qr-code ^2.0, qui lui-même exige
  PHP ^7.1 — incompatible avec PHP 8.3.14 utilisé par le projet. Il ne
  peut donc pas être installé aux côtés de Laravel 13.

  Le blocage s'est ensuite étendu à tout le projet : laravel/tinker était
  figé à "^2.9" dans composer.json, alors que seule la version 3.0.2 de
  Tinker supporte illuminate/support ^13.0 (compatible Laravel 13). Tant
  que cette contrainte n'était pas élargie, aucun "composer update" ne
  pouvait aboutir, quel que soit le package concerné.

CORRECTIONS APPLIQUEES (dans l'ordre) :

  a) Retrait définitif de simplesoftwareio/simple-qrcode de composer.json.

  b) Élargissement des contraintes bloquantes dans composer.json :
       "laravel/tinker"   : "^2.9|^3.0"
       "phpunit/phpunit"  : "^11.0|^12.0"   (dans require-dev)

  c) composer update -W  (résolution globale des dépendances)

  d) Installation du fork activement maintenu, compatible Laravel 13 /
     PHP 8.3 / bacon-qr-code ^3.0 :
       composer require f9webltd/simple-qrcode

  e) Dans la vue, remplacement de l'appel à la façade (dont
     l'enregistrement via auto-discovery posait encore souci) par une
     instanciation directe de la classe Generator du package, qui ne
     dépend d'aucun service provider :

       Avant :
         {!! \SimpleSoftwareIO\QrCode\Facades\QrCode::format('svg')
               ->size(200)->generate($commande->code_authenticite) !!}

       Après :
         {!! (new \F9WebLtd\QrCode\Generator)->format('svg')
               ->size(200)->generate($commande->code_authenticite) !!}

  f) Nettoyage des caches Laravel après chaque étape :
       php artisan config:clear
       php artisan view:clear

  Aucune autre partie du fichier n'a changé (affichage du statut, suivi de
  livraison, formulaire de confirmation de réception, formulaire d'avis,
  bouton d'annulation).

3. SEEDERS (database/seeders/)
-------------------------------------------------------------------------------

CONTEXTE :
  22 seeders + 1 DatabaseSeeder ont été générés pour couvrir l'ensemble des
  tables du schéma ElevConnect, avec au moins 4 lignes par table et le
  respect des contraintes de clés étrangères et des triggers métier MySQL
  (trg_annonces_type_role, trg_commandes_pas_auto_commande,
  trg_actualites_auteur_role).

FICHIERS LIVRES :

  VendeurAccessoireSeeder.php  VeterinaireSeeder.php
  LivreurSeeder.php            AdministrateurSeeder.php

  CommandeSeeder.php           PaiementSeeder.php

  ReductionLivraisonSeeder.php VersementSeeder.php
  ServiceVeterinaireSeeder.php RendezVousSeeder.php
  AbonnementSeeder.php         NotificationSeeder.php
  ActualiteSeeder.php          ActualiteMediaSeeder.php
  DatabaseSeeder.php (orchestre tous les seeders ci-dessus, dans cet ordre)

EVOLUTION SUITE AU CHANGEMENT DE SCHEMA (localisation déplacée) :
  Les migrations des 7 tables de profil (eleveurs, acheteurs,
  vendeur_provende, vendeur_accessoire, veterinaires, livreurs,
  administrateurs) ont été modifiées pour supprimer la colonne
  "localisation" (POINT/SRID 4326) : la géolocalisation (latitude,
  longitude en decimal) est désormais capturée une seule fois sur la
  table utilisateurs, commune à tous les rôles.

  En conséquence, les seeders ont été mis à jour :
    - UtilisateurSeeder.php : ajoute désormais 'latitude' et 'longitude'
      (decimal) à chaque utilisateur créé, avec des coordonnées cohérentes
      par ville (Parakou, Cotonou, Porto-Novo, Abomey).
    - EleveurSeeder.php, AcheteurSeeder.php, VendeurProvendeSeeder.php,
      VendeurAccessoireSeeder.php, VeterinaireSeeder.php, LivreurSeeder.php,
      AdministrateurSeeder.php : suppression complète du champ
      'localisation' et des tableaux de coordonnées DB::raw("ST_SRID(...)")
      qui n'ont plus lieu d'être, ces tables ne portant plus cette colonne.

  Tous les autres seeders (annonces, commandes, paiements, livraison,
  versements, services vétérinaires, rendez-vous, abonnements,
  notifications, actualités) restent inchangés : ils ne référençaient pas
  cette colonne.

COMMANDE DE LANCEMENT :
  php artisan migrate:fresh --seed
